## 一、阶段背景

当前 `main` 仍停留在 `1192525 feat: add one-click dev startup scripts`。已经完成并通过独立审查/测试的两个阶段成果仍在独立分支：

- `feat/a814802615/sprint-1-agent`：Sprint 1 自然语言解析闭环，提交 `74f355a`。
- `chore/a814802615/ui-branding`：UI 可见品牌迁移，提交 `db70eea`。

Sprint 2 文档明确写有前置条件：Sprint 1 代码已合并 main，`/api/agent/parse-task` 接口稳定可用。若继续直接从当前 `main` 开发 OCR/编排功能，会丢失 Sprint 1 Agent 前置能力，并重复处理 UI 品牌迁移。

因此本阶段先创建一个干净的累计开发基线：只集成已经通过 gate 的阶段提交，跑回归检查，为 Sprint 2 OCR 与日程编排开发提供一致基线。

## 二、目标与范围

### 目标

1. 将 Sprint 1 Agent 解析闭环提交 `74f355a` 集成到当前基线。
2. 将 UI 品牌迁移提交 `db70eea` 集成到同一基线。
3. 解决可能出现的 cherry-pick 冲突，保持两个阶段的行为与验收结论不回退。
4. 运行后端与前端关键检查，确认累计基线可继续用于后续 Sprint 2 开发。

### 范围内

- Cherry-pick 已通过审查/测试的两个阶段提交。
- 如有冲突，只做冲突合并所需的最小调整。
- 补充本阶段集成规划文档。
- 运行 `uv run ruff check .`、`uv run pytest`、`pnpm check`、`pnpm type-check`。

### 范围外

- 不新增 OCR、日程编排、文件导入等 Sprint 2 新功能。
- 不修改非冲突相关业务逻辑。
- 不处理 UI 品牌阶段遗留的 P2 文档/Agent 身份债务。

## 四、验收标准

1. `git log` 中能看到 `74f355a` 与 `db70eea` 对应阶段成果已进入当前分支历史。
2. Sprint 1 Agent 路由、服务、前端入口和测试文件存在且未回退。
3. UI 常规可见品牌仍为 BrightToDo，FreeToDo 只在关于来源说明或允许的元数据/内部兼容标识中保留。
4. `uv run ruff check .`、`uv run pytest`、`pnpm check`、`pnpm type-check` 均通过。
5. 独立审查 agent 与独立测试 agent 未发现 P0/P1 后，才保留本阶段提交状态。

## 五、风险与处理

| 风险 | 处理 |
|---|---|
| 两个阶段提交之间出现冲突 | 只做最小冲突合并，保留两个阶段已验收行为 |
| 回归检查暴露已有问题 | 先判断是否由集成引入；若是则修复后重新跑 gate |
| main 未直接合并阶段提交 | 本阶段产出可作为后续开发基线或后续 PR/合并候选 |
