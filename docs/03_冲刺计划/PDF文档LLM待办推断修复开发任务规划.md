# PDF 文档 LLM 待办推断修复开发任务规划

## 背景

当前文件导入链路已经支持 PDF/DOCX/TXT 等文件解析，但文本类文件主要依赖规则从行文本中寻找显式待办线索。真实用户上传的课程资料、作业说明、会议纪要、论文或合同通常不会直接写出“TODO/待办”，导致 PDF 上传后可能提示“文件已解析，但未识别到待办任务”。这与 BrightToDo 的 AI Agent 文件导入目标不一致。

## 目标

- 对文本型文件先提取可读文本，再在缺少显式待办线索时调用 LLM 推断用户需要执行的后续任务。
- LLM 输出需要包含标题、描述、优先级、初始执行时长估计、可选截止时间、来源文本与置信度。
- 当 LLM 不可用或调用失败时，保留本地规则回退能力，避免文件导入接口直接失败。
- 保持默认不落库，仍由用户在前端确认后再创建草稿待办。

## 实现范围

- 后端 `AgentImportService`
  - 新增文档 LLM 分析提示词与 JSON 响应解析。
  - 新增 LLM 输出字段校验和归一化。
  - 新增初始执行时长估算兜底规则。
  - 保留当前显式待办行的快速本地解析。
- 后端测试
  - 覆盖普通 PDF 通过 LLM 推断任务。
  - 覆盖 LLM 返回缺失时长时的默认时长估计。
  - 保持原有本地规则、非法文件、损坏 PDF/DOCX、超大文件和创建回滚测试。

## 验收标准

- 普通 PDF 文本中没有 TODO/待办字样时，可以由 LLM 返回待确认任务。
- 返回任务包含 `duration` 初始估计。
- LLM 不可用时原有显式待办文件仍可解析。
- `uv run python -m ruff check .` 通过。
- `uv run pytest` 通过。
- 独立审查 agent 与独立测试 agent 均 PASS 后提交。
